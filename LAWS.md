# LAWS.md — trios Constitutional Document

> **SUPREMACY**: This document is the supreme law of the trios repository.
> All code, commits, PRs, CI workflows, and agents MUST comply.
> Violations are grounds for immediate revert. No exceptions.

---

## CORE LAWS

### L1: NO .sh files
All automation must be Rust binaries (xtask) or Makefile. Shell scripts (`.sh`) are **banned**.

### L2: Every PR closes an issue
Every PR description MUST contain `Closes #N`. Orphan PRs are **banned**.

### L3: clippy zero warnings
```bash
cargo clippy -- -D warnings
```
Must pass before any merge. Zero tolerance.

### L4: Tests before merge
```bash
cargo test
```
All tests must pass. New code requires new tests.

### L5: Port 9005 is trios-server
The MCP server always runs on `0.0.0.0:9005`. Never change without a migration plan.

### L6: Fallback required for GB tools
`trios-gb` tools must return `Err` (not panic) if `gitbutler-cli` is not found.

### L7: Experience log
Every significant task writes a line to `.trinity/experience/`.

### L8: PUSH FIRST LAW
Every file change = immediate commit + push. No local-only changes.

## RING ARCHITECTURE LAWS

> These laws define the Trinity Ring system. Violating them corrupts the architecture.

### L9: BR-EXT Reference Integrity — NO PHANTOM IMPORTS

**THE LAW:** Any file referencing artifacts in `dist/` MUST verify those artifacts exist.
Before writing ANY `<script src>`, `import from`, or manifest reference — run:
```bash
ls crates/trios-ext/extension/dist/
```
If the file is not in the list → **DO NOT REFERENCE IT**.

**Current valid BR-EXT artifacts (as of v1 seal):**
```
dist/trios_ext_br.js       ← wasm-bindgen JS glue (39KB)
dist/trios_ext_br_bg.wasm  ← compiled WASM binary (239KB)
dist/bg-sw.js              ← MV3 service worker
dist/bootstrap.js          ← sidepanel bootstrap
dist/github-bootstrap.js   ← GitHub content script bootstrap
dist/claude-bootstrap.js   ← Claude content script bootstrap
```

**VIOLATION PATTERN (BANNED):**
```html
<!-- BANNED: trios_ext.js does not exist -->
<script src="dist/trios_ext.js"></script>
<!-- BANNED: trios_ext_bg.wasm does not exist -->
```

**CORRECT PATTERN:**
```html
<script type="module">
  import init from './dist/trios_ext_br.js';
  await init('./dist/trios_ext_br_bg.wasm');
</script>
```

### L10: Ring Seal Protocol

When a crate is declared **SEALED** (reference v1, v2, etc.):
1. A `SEAL.md` file is placed in the crate root with SHA hash and date
2. No agent may modify **Silver rings** (SR-XX, EX-XX) without explicit human approval
3. Only Bronze ring output files (`BR-*`) may be regenerated by build
4. The seal hash is verified by `laws-guard.yml` CI job

**Currently sealed:**
- `crates/trios-ext` → **v1** (2026-04-23) — Trinity Agent Bridge reference implementation

### L11: No Phantom Imports

An agent must NEVER reference a file path it did not verify exists.
This applies to: HTML `<script src>`, JS `import from`, manifest JSON paths, Cargo `path = ...`.

**Verification ritual before any path reference:**
1. `ls` the directory
2. Confirm the exact filename (including suffix like `_br`, `_bg`)
3. Only then write the reference

## RING ISOLATION LAWS (R1–R5)

### R1: Ring does not import from neighboring ring directly
Only via `Cargo.toml` path dependency. No `use crate::other_ring::*`.

### R2: Each ring is a separate Cargo package
Every ring directory = separate `Cargo.toml` with unique `[package]`.

### R3: Every ring directory must have RING.md
Format: purpose, API surface, dependencies, invariants.

### R4: No cross-ring file placement
A Silver ring file (e.g. `dom.rs`) CANNOT be placed in a Bronze ring directory.
A Bronze artifact CANNOT be placed in a Silver source directory.

### R5: Sharing only via path dependency
```toml
trios-ext-silver = { path = "../trios-ext/rings/silver" }
```
Never copy code between rings. Extract to a shared ring if needed.

## GOLD / SILVER / BRONZE RING DEFINITION

```
🥇 GOLD RINGS    = crates/     (top-level Cargo workspace members)
                   Each crate is ONE gold ring.
                   Examples: trios-ext, trios-http, trios-a2a, trios-server

🥈 SILVER RINGS  = rings/SR-XX or rings/EX-XX inside each crate
                   Business logic, Rust source code, algorithms.
                   Immutable once sealed. Change requires explicit approval.
                   Examples: EX-00 (WASM entry), EX-01 (DOM bridge),
                             SR-00 (Identity), SR-01 (Transport)

🥉 BRONZE RINGS  = rings/BR-* inside each crate
                   OUTPUT ONLY. Compiled artifacts, dist/, HTML shells.
                   Generated by build — never hand-edited after seal.
                   Examples: BR-EXT (WASM bundle), BR-APP (dx build output),
                             BR-OUTPUT (binary artifacts)
```

**INVARIANT:** Bronze rings contain no business logic. Silver rings contain no build artifacts.
Violating this is a critical architecture error.
