## 📌 Контекст и Происхождение

**«The Power of 10: Rules for Developing Safety-Critical Code»** — документ, написанный Жераром Хольцманном (Gerard J. Holzmann) в 2006 году для NASA Jet Propulsion Laboratory (JPL). Исходно ориентирован на язык **C**, но принципы универсальны и напрямую применимы к нашему стеку (**Rust**, **Zig**).

### Почему NASA создала именно 10 правил?

Большинство существующих coding guidelines содержат **100+ правил** с сомнительным обоснованием и не поддаются автоматической проверке. Холцманн поставил условие: правил не больше десяти, каждое — автоматически проверяемо статическим анализатором, каждое — с чёткой инженерной мотивацией.

> *«If these rules seem draconian at first, bear in mind that they are meant to make it possible to check safety-critical code where human lives can very literally depend on its correctness. The rules are like the seat belts in a car.»*
> — Gerard J. Holzmann, JPL

### Rule 1 — Простой Поток Управления

**Оригинал:**
> Restrict all code to very simple control flow constructs — do not use `goto` statements, `setjmp` or `longjmp` constructs, or direct or **indirect recursion**.

**Обоснование:**
- `goto`, `setjmp/longjmp` делают граф потока управления непредсказуемым для статических анализаторов
- Запрет рекурсии создаёт **ациклический граф вызовов** (DAG) → доказуема конечность стека
- Статический анализ может доказать ограниченность стека и выполнения

**Нарушение в нашем коде — красные флаги:**
```rust
// ❌ ЗАПРЕЩЕНО: рекурсия
fn compute_bpb(depth: usize) -> f64 {
    if depth == 0 { return 2.5329; }
    compute_bpb(depth - 1) // ← рекурсия = нарушение Rule 1
}

// ✅ ПРАВИЛЬНО: итерация с явной границей
fn compute_bpb(max_depth: usize) -> f64 {
    let mut result = 2.5329;
    for _i in 0..max_depth { // ← Rule 2: фиксированная граница
        result = update_step(result);
    }
    result
}
```

**Применение к IGLA RACE:**
- `run_worker()` в `asha.rs` — итерационный loop, не рекурсивный ✅
- ASHA rungs: `for &rung in &[1000u64, 3000, 9000, 27000]` — фиксированный массив ✅
- Запрет рекурсии в Rust проще: компилятор заставляет явно боксировать рекурсивные типы

### Rule 2 — Фиксированные Границы Циклов

**Оригинал:**
> Give all loops a fixed upper bound. It must be trivially possible for a checking tool to prove statically that the loop cannot exceed a preset upper bound on the number of iterations.

**Обоснование:**
- Совместно с Rule 1 делает impossible «runaway code» — бесконечные зависания
- Нетерминирующие итерации (scheduler, event loop) — исключение: инструмент должен доказать, что они **не могут** завершиться (обратное правило)

**Нарушение:**
```rust
// ❌ ПЛОХО: граница зависит от внешних данных, не проверяема статически
while neon_has_more_trials() {
    process_trial();
}

// ✅ ХОРОШО: явная верхняя граница
const MAX_TRIALS: usize = 100_000;
for _i in 0..MAX_TRIALS {
    if !neon_has_more_trials() { break; }
    process_trial();
    // assert!(_i < MAX_TRIALS - 1, "trial overflow"); // ← Rule 5
}
```

**Применение к IGLA RACE:**
- ASHA rungs — массив из 4 элементов: `[1000, 3000, 9000, 27000]` ✅
- `run_worker()` loop — ИСКЛЮЧЕНИЕ (нетерминирующий event-loop по правилу)
- Но внутренние циклы тренировки ОБЯЗАНЫ иметь `MAX_STEPS` guard

**Критическое место:**
```rust
// IGLA trainer: обязательная граница
const MAX_TRAINING_STEPS: u64 = 100_000; // guard сверх любого rung
for step in 0..steps.min(MAX_TRAINING_STEPS) { ... }
```

### Rule 3 — Нет Динамической Памяти после Инициализации

**Оригинал:**
> Do not use dynamic memory allocation after initialization.

**Обоснование:**
- `malloc`/GC имеют **непредсказуемое поведение** и латентность
- Целый класс ошибок: use-after-free, double-free, фрагментация кучи, OOM mid-flight
- Фиксированная pre-allocated область памяти → доказуемые resource bounds

**В контексте Rust:**
Rust не устраняет проблему полностью — `Vec::push()` вызывает `realloc`. NASA-совместимый Rust:

```rust
// ❌ ПЛОХО: runtime heap allocation
fn collect_bpb_history() -> Vec<f64> {
    let mut v = Vec::new();  // alloc
    v.push(2.5329);          // possible realloc
    v
}

// ✅ ХОРОШО: pre-allocated / fixed-size
const MAX_HISTORY: usize = 10_000;
fn collect_bpb_history(buf: &mut [f64; MAX_HISTORY], len: &mut usize) {
    // stack allocation, zero heap
    buf[*len] = 2.5329;
    *len += 1;
}

// ✅ ХОРОШО в Rust: Vec::with_capacity() при инициализации
fn init_trial_buffer() -> Vec<f64> {
    Vec::with_capacity(10_000) // alloc только при init, не в runtime-loop
}
```

**Применение к IGLA RACE:**
- Model weights (`Vec<f32>`) — инициализация один раз, не в hot loop ✅
- Neon connection pool — создаётся при запуске, переиспользуется ✅
- `String::new()` в hot loop при парсинге BPB — нарушение, нужен `&str` / fixed buffer

### Rule 4 — Функции ≤ 60 строк (1 лист бумаги)

**Оригинал:**
> No function should be longer than what can be printed on a single sheet of paper in a standard format with one line per statement and one line per declaration. Typically, this means no more than about **60 lines of code** per function.

**Обоснование:**
- Функция = логическая единица, верифицируемая как единое целое
- Длинные функции → признак плохой структуры → невозможность unit-testing
- «Мозг не может удержать > 7 ± 2 вещей одновременно» (Miller, 1956)

**Применение к IGLA RACE — нарушения:**
```
crates/trios-train-cpu/src/bin/tjepa_train.rs — 309 строк!
```

Этот файл **нарушает Rule 4**. Нужна декомпозиция:

```rust
// ❌ СЕЙЧАС: один main() на 200+ строк
pub fn main() { /* 250 lines of training logic */ }

// ✅ ДОЛЖНО БЫТЬ:
fn parse_args() -> TrainConfig { /* ≤ 20 lines */ }

fn run_asha_rung(model: &mut NgramEncoder, rung: u64) -> f64 { /* ≤ 40 lines */ }

fn main() {

    print_result(bpb, BASELINE_BPB);
}
```

### Rule 5 — Минимум 2 Assertion на Функцию

**Оригинал:**
> The code's assertion density should average to minimally **two assertions per function**. Assertions must be side-effect free and defined as Boolean tests. When an assertion fails, an explicit recovery action must be taken.

**Обоснование:**
- Промышленная статистика: unit tests находят 1 дефект на каждые 10–100 строк кода
- Assertions как **исполняемая документация** инвариантов
- Pre/postconditions функций, loop invariants, parameter validity
- Side-effect free → можно отключить в performance-critical release

**Применение к IGLA RACE:**
```rust
// ✅ ПРАВИЛЬНО: assertions в каждой функции
pub fn parse_bpb(stdout: &[u8]) -> anyhow::Result<f64> {
    let text = std::str::from_utf8(stdout)?;
    // Assertion 1: входные данные не пустые
    assert!(!text.is_empty(), "trainer stdout is empty — L-R8 violation");
    
    let last_line = text.lines().last()
        .ok_or_else(|| anyhow::anyhow!("no output lines"))?;
    
    // Assertion 2: формат соблюдён
    assert!(last_line.starts_with("BPB="),
        "stdout last line must start with 'BPB=', got: {:?}", last_line);
    
    let bpb: f64 = last_line[4..].trim().parse()?;
    
    // Assertion 3: физически осмысленное значение
    assert!(bpb > 0.0 && bpb < 20.0,
        "BPB out of valid range [0, 20]: {}", bpb);
    
    Ok(bpb)
}
```

**Карта assertions для IGLA RACE:**
| Функция | Assertion 1 | Assertion 2 | Assertion 3 |
|---------|-------------|-------------|-------------|
| `parse_bpb()` | stdout не пустой | starts_with("BPB=") | bpb ∈ (0, 20) |
| `run_worker()` | machine_id не пустой | workers ≥ 1 | rung ∈ {1k,3k,9k,27k} |
| `should_prune()` | threshold > 0 | rung валиден | bpb > 0 |
| `register_trial()` | neon_url не пустой | config не None | trial_id > 0 (post) |
| `sample_config()` | d_model ∈ {128,192,256,384} | lr > 0 | seed ∈ [40,50] |

### Rule 6 — Минимальная Область Видимости Данных

**Оригинал:**
> Declare all data objects at the **smallest possible level of scope**.

**Обоснование:**
- Если объект вне scope → другой модуль не может его испортить
- Минимальный scope → минимальное число мест, где значение могло измениться
- Предотвращает reuse переменных для несовместимых целей

**Применение к IGLA RACE:**
```rust
// ❌ ПЛОХО: глобальный mutable state
static mut BEST_BPB: f64 = f64::MAX; // ← Rule 6 + thread safety violation

// ✅ ХОРОШО: scope-limited shared state
pub async fn run_workers(n: usize) {
    let best_bpb = Arc::new(RwLock::new(f64::MAX)); // scope = эта функция
    let handles: Vec<_> = (0..n).map(|id| {
        let best = Arc::clone(&best_bpb);
        tokio::spawn(async move { run_worker(id, best).await })
    }).collect();
    // best_bpb не покидает эту функцию
}
```

**Rust преимущество:** ownership system принуждает к Rule 6 структурно.
`let` bindings создают scope, borrow checker предотвращает утечку scope.

### Rule 7 — Проверка Возвращаемых Значений

**Оригинал:**
> Each calling function must **check the return value** of nonvoid functions, and each called function must check the validity of all parameters provided by the caller.

**Обоснование:**
- Наиболее часто нарушаемое правило
- Необработанные ошибки = молчаливая деградация (worst case: wrong BPB в Neon)
- Даже `printf()` и `close()` должны проверяться (или явно cast в `void`)

**Применение к IGLA RACE:**
```rust
// ❌ ПЛОХО: игнорирование ошибок
let _ = update_neon_rung(url, trial_id, rung, bpb).await;
// Если Neon упал → результат потерян, никто не знает

// ✅ ХОРОШО: явная обработка
update_neon_rung(url, trial_id, rung, bpb).await
    .unwrap_or_else(|e| {
        // fallback: write to .trinity/experience/ (Law L-R3)
        log_to_file(trial_id, rung, bpb, &e.to_string());
        eprintln!("WARN: Neon write failed, fallback used: {e}");
    });
```

**Rust преимущество:** `Result<T, E>` + `#[must_use]` + компилятор предупреждает об unused Result.
Наш **L-R5** (`cargo clippy -D warnings`) автоматически ловит нарушения Rule 7.

### Rule 8 — Ограниченное Использование Препроцессора

**Оригинал:**
> The use of the preprocessor must be limited to the inclusion of header files and simple macro definitions. Token pasting, variable argument lists (ellipses), and recursive macro calls are not allowed.

**Обоснование:**
- C препроцессор = «инструмент обфускации»
- Сложные макросы обходят систему типов и статические анализаторы
- `#ifdef` с N директивами → 2^N версий кода для тестирования

**Применение к IGLA RACE (Rust):**
В Rust нет C-препроцессора, но есть `macro_rules!` и `proc_macro`:

```rust
// ❌ ПЛОХО: сложный macro_rules с рекурсией
macro_rules! compute_loss {
    ($x:expr) => { compute_loss!($x, 1.0) }; // recursive
    ($x:expr, $w:expr) => { $x * $w };
}

// ✅ ХОРОШО: простые const/fn вместо macro
const JEPA_WEIGHT: f64 = 0.25;
fn compute_loss(bpb: f64, weight: f64) -> f64 { bpb * weight }

// ✅ ДОПУСТИМО: простые assert-макросы (side-effect free, Rule 5)
assert!(bpb > 0.0);
debug_assert!(d_model >= 256, "GF16 requires d_model >= 256 (Law L-R9)");
```

**Conditional compilation в Rust:**
```rust
// ❌ МНОГО #[cfg]: сложно тестировать
#[cfg(feature = "gf16")] fn forward_gf16() { ... }
#[cfg(not(feature = "gf16"))] fn forward_gf16() { ... }

// ✅ ЛУЧШЕ: runtime dispatch, один код path
fn forward(precision: Precision) -> f64 {
    match precision {
        Precision::F32 => forward_f32(),
        Precision::Gf16 => forward_gf16(), // один path, полностью тестируем
    }
}
```

### Rule 9 — Ограниченное Использование Указателей

**Оригинал:**
> The use of pointers must be restricted. Specifically, no more than **one level of dereferencing** should be used. Pointer dereference operations may not be hidden in macro definitions or inside typedef declarations. **Function pointers are not permitted.**

**Обоснование:**
- Указатели — наиболее частый источник ошибок, даже у опытных программистов
- Цепочки `**ptr` → невозможен автоматический анализ data flow
- Function pointers блокируют доказательство отсутствия рекурсии

**Применение к IGLA RACE (Rust):**
Rust references — безопасная альтернатива указателям, borrow checker = статический верификатор Rule 9.

```rust
// ❌ В C нарушение: double dereference, function pointers
// int **matrix; (*fn_ptr)(x);

// ✅ В Rust — аналог через типы
// Один уровень: &T, &mut T, Box<T> — OK
let model: &NgramEncoder = &encoder;        // один уровень ✅
let weight: &f32 = &model.weights[0];       // один уровень ✅

// ❌ Два уровня через Box<Box<T>> — подозрительно
let nested: Box<Box<Vec<f32>>> = ...; // спрашивай: зачем?

// Function pointers — Rust trait objects как альтернатива
// ❌ fn_ptr style:
type TrainFn = fn(config: &Config) -> f64; // function pointer

// ✅ Trait-based dispatch (статически верифицируем)
trait Trainer { fn train(&self, cfg: &Config) -> f64; }
struct NgramTrainer; impl Trainer for NgramTrainer { ... }
struct JepaTrainer;  impl Trainer for JepaTrainer  { ... }
```

### Rule 10 — Нулевые Предупреждения Компилятора

**Оригинал:**
> All code must be compiled, from the first day of development, with **all compiler warnings enabled at the most pedantic setting** available. All code must compile without warnings. All code must also be checked daily with at least one strong **static source code analyzer** and should pass with zero warnings.

**Обоснование:**
- Нет оправдания не использовать статический анализ — инструменты доступны и точны
- «Zero warnings» — даже если компилятор «ошибается»: путающий код надо переписать
- Early и часто: не накапливать warnings

**Применение к IGLA RACE:**

## 💡 Rust vs C: Структурные Гарантии

Rust даёт **структурные** (compile-time) гарантии для 6 из 10 правил NASA:

| Rule | C (manual) | Rust (structural) |
|------|------------|-------------------|
| Rule 1: No recursion | Ручная проверка | Clippy: `#[warn(unconditional_recursion)]` |
| Rule 3: Memory safety | Ручной `free()` | Ownership system — compile-time |
| Rule 6: Scope | Ручная дисциплина | Borrow checker — compile-time |
| Rule 7: Error handling | Игнорируем `errno` | `Result<T,E>` + `#[must_use]` |
| Rule 9: Pointer safety | Ручной анализ | Borrow checker, no dangling ptrs |
| Rule 10: Zero warnings | Настройка gcc | `cargo clippy -D warnings` (L-R5) |

**Вывод:** Rust переводит 60% NASA правил из «дисциплины» в «compile-time error».
Именно поэтому L-R1 («RUST ONLY») — не произвол, а инженерное решение.
